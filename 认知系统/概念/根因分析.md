# 根因分析

## 一句话定义

根因分析是一种系统性地追溯问题因果链直至找到根本原因的方法论——不停留在"表面症状"，而是持续追问"为什么"直到触及可被改变的系统性原因，从而避免问题反复发生。

## 提出者与背景

- **5 Whys 创始人**：大野耐一（Taiichi Ohno），丰田生产方式（TPS）的缔造者，"连问五次为什么，问题的本质和解决办法就会变得明朗"
- **鱼骨图创始人**：石川馨（Kaoru Ishikawa），日本质量管理先驱，发明因果图（石川图/鱼骨图）
- **人为失误分析**：James Reason 的"瑞士奶酪模型"将根因分析从机械故障扩展至组织因素
- **针对的问题**：大多数问题解决只处理了"表面原因"（symptom），问题会反复出现——只有找到根因才能永久消除问题

## 核心要义

### 5 Whys 方法

通过连续追问"为什么"来层层剥开因果链：

```
问题：客户投诉产品交付延迟
  │
  Why 1: 为什么延迟？→ 因为生产排期被推迟
  │
  Why 2: 为什么排期被推迟？→ 因为原材料到货晚了
  │
  Why 3: 为什么原材料到货晚了？→ 因为采购订单下得晚
  │
  Why 4: 为什么采购订单下得晚？→ 因为库存预警系统没有及时触发
  │
  Why 5: 为什么预警系统没触发？→ 因为库存阈值的设定基于过时的需求预测
  │
  ▼ 根因：需求预测更新机制缺失 → 建立周期性需求复盘流程
```

### 鱼骨图（石川图）

从多个维度分类潜在原因：

```
                        问题/结果
                         ▲
                        │
    人员    方法    机器    材料    环境    测量
  (Man)  (Method) (Machine)(Material)(Environment)(Measurement)
    │       │       │       │       │       │
    ├─...   ├─...   ├─...   ├─...   ├─...   ├─...
    │       │       │       │       │       │
```

### 根因的三个层次

| 层次 | 描述 | 解决效果 |
|------|------|----------|
| **直接原因** | 最表层的触发因素 | 只能暂时缓解症状 |
| **促成原因** | 使直接原因得以发生的条件 | 减少问题频率 |
| **根本原因** | 系统性缺陷，使问题可能反复发生 | 永久消除问题 |

### 关键论点

1. **5 不是魔数**："五次"是经验性的，实际可能需要 3 次或 8 次——关键是到达可被改变的系统性原因
2. **避免"人的失误"陷阱**：将根因归结为"某某人犯了错"是懒惰的分析——要追问"系统为什么允许他犯错"
3. **多条因果链**：一个问题通常有多条因果链汇合（瑞士奶酪模型），需要同时阻断多条链
4. **纠正措施 vs 预防措施**：根因分析的目标是预防措施（消除根因），而非纠正措施（修补后果）
5. **证据驱动**：每个"为什么"的回答都应有事实支撑，而非猜测

## 常用工具对比

| 工具 | 优势 | 局限 | 适用场景 |
|------|------|------|----------|
| **5 Whys** | 简单快速，不需要培训 | 可能停在表面、只追一条链 | 日常问题快速诊断 |
| **鱼骨图** | 多维度穷举、团队共创 | 容易变成"头脑风暴"而缺乏深度 | 多因素问题的初始分析 |
| **故障树分析（FTA）** | 严格的逻辑结构 | 需要专业知识，耗时 | 高风险系统（航空、核能） |
| **FMEA** | 前瞻性（故障前分析） | 复杂度高 | 产品设计阶段 |

## 通俗用法 vs 学术原义

| 维度 | 通俗用法 | 学术原义 |
|------|----------|----------|
| "找到根本原因" | "弄清楚为什么" | 系统性地追溯因果链至可被干预的系统性缺陷 |
| "问五个为什么" | "多问几个为什么" | 大野耐一的经验法则——追问至因果链的根节点 |
| "根因" | "最深的原因" | 消除后问题不再发生的最小充分原因集 |

## 代表思想家

- **大野耐一（Taiichi Ohno）** — 丰田生产方式创始人，5 Whys 的发明者
- **石川馨（Kaoru Ishikawa）** — 因果图（鱼骨图）的发明者，将质量管理从专家工具变为全员工具
- **James Reason** — 瑞士奶酪模型，将根因分析从机械故障扩展至组织因素
- **Sidney Dekker** — "Just Culture"理论——从"谁犯了错"转向"系统为什么允许犯错"

## 应用场景

- **制造业**：生产线故障的根因追溯——丰田生产方式的核心实践
- **IT 运维**：线上事故的 Post-mortem / RCA——不停在"服务器挂了"，追问为什么挂
- **医疗**：不良事件的根本原因分析（RCA），避免将医疗事故归结为个人失误
- **航空**：飞行事故调查——NTSB 的调查本质上就是根因分析
- **软件开发**：Bug 的根因分析——不是修 bug，而是追问"为什么这个 bug 能被引入"
- **个人成长**：反复犯的同一个错误背后，往往有一个未被触及的系统性根因

## 常见误读

- **误读 1**："问够五个为什么就行了" — 五次是经验数，关键是到达可改变的根因；可能 3 次就够，也可能需要 10 次
- **误读 2**："根因只有一个" — 复杂问题通常有多个根因汇合，需要同时解决
- **误读 3**："根因分析就是找谁负责" — 它的目的是改进系统，不是追责个人（归咎个人是分析的失败）
- **误读 4**："5 Whys 适合所有问题" — 它最适合中等复杂度的单一问题；对于多因素交织的复杂问题，需要鱼骨图或故障树
